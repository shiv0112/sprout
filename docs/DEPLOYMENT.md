# Deploying Kiln — Backend on Render, Frontend on Vercel

This describes the split deployment:

- **Backend** — all five Python microservices run in **one** Render Docker
  service, behind a Caddy reverse proxy.
- **Frontend** — `packages/registry_ui` (Next.js 16) runs on **Vercel**.

```
                         ┌──────────────────────────── Render (one web service) ─────────────┐
  Browser ── HTTPS ─►    │  Caddy  :$PORT                                                     │
      │                  │    ├─ /registry/*  ─► 127.0.0.1:8766  registry_api                 │
      │                  │    ├─ /chat/*      ─► 127.0.0.1:8765  chat_backend  (SSE)          │
   Vercel (Next.js) ─►   │    ├─ /synthesis/* ─► 127.0.0.1:8002  synthesis_service (SSE)      │
   /api/* route handlers │    ├─ /mcp/*       ─► 127.0.0.1:8768  mcp_server                   │
                         │    ├─ /executor/*  ─► 127.0.0.1:8767  tool_executor                │
                         │    └─ /health      ─► 200                                          │
                         │  supervisord runs all 5 services + Caddy                           │
                         └──── Render Postgres ────────── Render Key Value (Redis) ───────────┘
```

Files that make this work:

| File | Purpose |
|---|---|
| `docker/Dockerfile.render` | Single image: Python venv (all services) + Node/OpenCode + Caddy + supervisord |
| `docker/render/supervisord.conf` | Runs the 5 services + Caddy, logs to stdout |
| `docker/render/Caddyfile` | Path-prefix routing on `$PORT` |
| `render.yaml` | Render Blueprint: web service + Postgres + Redis + env wiring |
| `packages/registry_ui/.env.production.example` | Vercel env template |

---

## 1. Deploy the backend to Render

1. Push this repo to GitHub (already at `github.com/shiv0112/kiln`).
2. Render Dashboard ▸ **New ▸ Blueprint** ▸ select this repo. Render reads
   `render.yaml` and provisions:
   - `kiln-backend` (Docker web service, **plan: standard / 2 GB** — see sizing note)
   - `kiln-postgres` (PostgreSQL)
   - `kiln-redis` (Key Value / Redis)
   `DATABASE_URL`, `REDIS_URL`, and all inter-service URLs are wired automatically.
3. Fill in the secrets marked `sync: false` under the service's **Environment**:
   | Var | Value |
   |---|---|
   | `NVIDIA_API_KEY` | NVIDIA NIM key — **primary** for planning + agent execution (GLM-5.2) |
   | `MISTRAL_API_KEY` | Mistral key — **fallback** for planning/execution + synthesis (Codestral) |
   | `CLERK_DOMAIN` | e.g. `your-app.clerk.accounts.dev` |
   | `CLERK_SECRET_KEY` | Clerk secret key |
   | `CORS_ORIGINS` | your Vercel URL, e.g. `https://kiln.vercel.app` (comma-sep for several) |

   `KILN_INTERNAL_SECRET` is generated by Render automatically.
   `KILN_MCP_ISSUER_URL` is **auto-derived** from Render's `RENDER_EXTERNAL_URL`
   (+ `/mcp`) — set it only to override with a custom domain.
4. Deploy. When live, note the host (e.g. `kiln-backend.onrender.com`) and confirm:
   ```bash
   curl https://<render-host>/health          # -> ok
   curl https://<render-host>/registry/readyz # -> {"status":"ok",... "database":"ok"}
   curl https://<render-host>/registry/tools  # -> tool catalog
   ```

> **`CORS_ORIGINS` is mandatory in production.** The shared CORS installer
> raises on startup if it's unset when `KILN_ENV=production`. The browser calls
> `registry_api` and `chat_backend` directly (no Vercel rewrite), so their origin
> allowlist must include the Vercel URL.

## 2. Deploy the frontend to Vercel

1. Vercel ▸ **Add New ▸ Project** ▸ import this repo.
2. **Root Directory:** `packages/registry_ui` (this is a monorepo — required).
   Framework preset auto-detects **Next.js**; leave build/output defaults.
3. Add the environment variables from
   `packages/registry_ui/.env.production.example`, substituting your Render host.
   The `NEXT_PUBLIC_*` values **must include the path prefix** (`/registry`,
   `/chat`, …) since Caddy routes on it.
4. Deploy. Then go back and set the backend's `CORS_ORIGINS` to the Vercel URL
   (and redeploy the backend so it picks up the value).

> `next.config.ts` has `output: "standalone"` — harmless on Vercel (ignored).
> Middleware lives in `src/proxy.ts` (Next 16 convention), not `middleware.ts`.

## 3. Test the backend image locally (optional but recommended)

```bash
# Build the single-container backend
docker build -f docker/Dockerfile.render -t kiln-backend .

# Run it (uses your local .env; needs Postgres/Redis reachable, or set
# KILN_ENV=dev to use the SQLite fallback and skip the leader lock)
docker run --rm -p 8080:8080 -e PORT=8080 -e KILN_ENV=dev \
  -e KILN_DB_AUTO_CREATE=true --env-file .env kiln-backend

curl localhost:8080/health
curl localhost:8080/registry/tools
```

## Notes & caveats

- **Sizing:** five Python services in one container need real RAM. The 512 MB
  free/starter instances will OOM — use `standard` (2 GB) or larger. If you want
  a smaller footprint, drop services you don't need from `supervisord.conf` and
  the `Caddyfile` (e.g. omit `mcp_server` or `synthesis_service`).
- **Redis:** `chat_backend` requires `REDIS_URL` when `KILN_ENV=production` (for
  its single-writer leader lock). The blueprint provisions it. To skip Redis
  entirely on a single instance, set `KILN_CHAT_LEADER_LOCK=false` instead.
- **Database schema:** `KILN_DB_AUTO_CREATE=true` creates tables on boot. The
  production-grade alternative is to run `kiln-registry migrate` (Alembic) once —
  e.g. as a Render pre-deploy command — and set `KILN_DB_AUTO_CREATE=false`.
- **LLM providers:** planning + agent execution use **NVIDIA NIM (GLM-5.2) as
  primary and Mistral as fallback** — both OpenAI-compatible. If NVIDIA
  rate-limits or errors, requests fall through to Mistral automatically (the
  planner walks the chain with backoff; the AG2 executor uses a multi-entry
  `config_list`). Tune with `KILN_LLM_PRIMARY`, `KILN_NIM_MODEL`,
  `KILN_MISTRAL_MODEL`. The NIM model must support OpenAI **tool-calling**
  (GLM-5.2 and Llama-3.3-70B do).
- **Synthesis:** runs the OpenCode CLI (Node) as a subprocess — no Docker-in-Docker,
  so it works on Render. Defaults to Mistral **Codestral** (code-specialized).
  To move it off Mistral too, set `KILN_SYNTHESIS_OPENCODE_MODEL=nim/z-ai/glm-5.2`
  (the `nim` provider is already defined in `opencode.json`).
- **MCP OAuth:** the MCP endpoint is reachable at `https://<render-host>/mcp`.
  The OAuth issuer is auto-derived from `RENDER_EXTERNAL_URL` (+ `/mcp`), so the
  discovery/callback flow works out of the box. Override `KILN_MCP_ISSUER_URL`
  only if you serve MCP from a custom domain.
- **Free-tier cold starts:** Render free/idle services sleep; the first request
  after idle will be slow while the container wakes.
